## 解析してほしい内容

### 0) 全体像の把握（最初に必ず）

* Notebookの狙い（ベースライン/特徴量/チューニング/アンサンブル等）を要約
* データ読み込み〜提出までの流れを「工程別」に整理（EDA→前処理→学習→CV→推論→submission）
* 使っているCV戦略（StratifiedKFold等）と、その妥当性（AUC前提、層化、リーク耐性）を評価

### 1) 再現性・リーク・分割の危険点チェック

* seed固定、乱数、CPU/GPU差、依存ライブラリ差
* train/test混線、fit/transform順序の誤り、リーク特徴量の可能性
* もし時系列/グループっぽい列があれば「分割単位が適切か」を指摘

### 2) 前処理・特徴量のレビュー（S6E2想定）

* 欠損処理、外れ値、カテゴリ処理（OneHot/Target Encoding/CatBoostなど）の適合性
* `id` の扱い（学習に入れていないか、ただし並び確認には使っているか）
* スケーリングの要否（線形/NN系 vs GBDT系で過不足がないか）
* 相関の高い列、情報量の薄い列の扱い（削除/正則化/木モデルに任せる等）

### 3) モデル設計レビュー（AUC最適化）

* モデル選定（LightGBM/XGBoost/CatBoost/NN/LogReg等）と理由
* 目的関数/評価関数がAUCに整合しているか（early stoppingのmonitor含む）
* クラス不均衡がある場合の扱い（class_weight / scale_pos_weight 等）
* 学習曲線・過学習兆候・fold間のブレの原因分析

### 4) 評価設計（LBブレ対策）

* CVスコアとPublic LBの乖離が出そうな要因
* OOFの作り方が正しいか（fold外予測の混入がないか）
* 必要なら「より堅い検証」（反復CV、seedアンサンブル、特徴量選択のCV内完結）を提案

### 5) 推論・提出物チェック

* 提出ファイルの列名・行数・順序の整合（`sample_submission`準拠）
* 出力が確率になっているか（0/1のラベル出力になってないか）
* 後処理（クリップ、rank平均、ブレンド、スタッキング）が妥当か

### 6) 改善提案（具体策）

以下の3段階で、**「どこをどう変えるか」まで**具体的に：

* **即効（〜30分）**：分割・seed・基本パラメータ・欠損処理・カテゴリ処理の最小改善
* **中規模（半日）**：Optuna/特徴量選択/安定化（反復CV・OOFベースのブレンド等）
* **本格（1〜数日）**：スタッキング、NN + GBDTブレンド、外部データ活用の可否検討（合成データ前提で慎重に）

### 7) バグ/危険箇所リスト

* 実行時エラーになり得る点（列名、型、欠損、カテゴリ未知値など）
* 計算量が重い/無駄な箇所
* 「偶然スコアが良く見える罠」（リーク、CV不備、評価の取り違え）
